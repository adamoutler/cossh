# QA Report: SSH-143 - Telnet No Text Input Issue

## Issue Description
Users reported that when connecting to a Telnet server, no text input was appearing on the screen. The connection was successful, but the terminal was unresponsive to keystrokes.

## Root Cause Analysis
1. **NetworkOnMainThreadException / UI Blocking**: The `TerminalSession` in `TerminalScreen.kt` was writing bytes directly to the `ptyOutputStream`, which was a direct wrapper around `tc.outputStream`. This caused silent drops of keystrokes as the Android system prevents network IO on the main thread, or caused the UI to lag significantly.
2. **Missing Remote Echo Negotiation**: The Apache Commons Net `EchoOptionHandler` was configured with `initremote = false`. This meant the CoSSH app was not asking the remote Telnet server to echo characters back to the client. Since `TerminalView` relies entirely on remote echo, characters typed were not displayed.
3. **Missing CRLF Translation**: When the UI blocking was temporarily fixed in earlier development, the `autoFlushingStream` was no longer translating raw carriage returns (`\r`) sent by the `TerminalView` into proper Telnet Network Virtual Terminal (NVT) `\r\n` (CRLF) sequences, causing real Telnet servers to ignore commands because they never saw an end-of-line.
4. **Buffered Stream Available() Issue**: The `PtyStreamBridge` relied on `inputStream.available() > 0` to check for incoming data before performing a blocking read, to supposedly respect coroutine cancellation. However, `TelnetInputStream` and buffered network sockets often return `0` for `available()` even when data is arriving, causing the stream to drop characters or freeze.

## Fix Implementation
- **Coroutine Channel Bridge**: Introduced a `kotlinx.coroutines.channels.Channel<ByteArray>` in `SshConnectionManager.kt`. When `ptyOutputStream.write()` is called from the UI, bytes are safely placed into this non-blocking channel. A dedicated `Dispatchers.IO` coroutine consumes this channel and executes the blocking `tc.outputStream.write()`.
- **OptionHandler Updates**: Updated `EchoOptionHandler` to `(false, true, false, true)` to explicitly request `IAC DO ECHO` from the server.
- **CRLF Translation reinstated**: Re-added a robust state machine to `autoFlushingStream` that safely intercepts `\r` and `\n` and translates them into explicit `\r\n` bytes sent to the Telnet server.
- **Blocking Reads**: Removed the `available() > 0` check in `PtyStreamBridge.kt` and reverted to standard blocking `inputStream.read(buffer)` running safely in an IO dispatcher.
