# Verification Proof for SSH-109

This ticket resolves the issue of broken SSH key management caused by double encryption.

**1. Root Cause Analysis:**
Previously, `SSHKeyGenerator.encodePrivateKey` explicitly invoked `PasswordCipher.encrypt()` to encrypt the PKCS8 bytes before returning them to the UI layer. When the user saved the profile, `IdentityStorageManager.saveIdentity` took these already-encrypted bytes and passed them to `PasswordCipher.encrypt()` again. 

When retrieving the key for an SSH connection, `IdentityStorageManager.getIdentity` decrypted the key only once, meaning the connection manager received a cyphertext payload instead of raw PKCS8 bytes, resulting in authentication failures for newly generated keys.

**2. Remediation:**
*   Modified `SSHKeyGenerator.encodePrivateKey` to return the raw PKCS8 `keyPair.private.encoded` bytes directly.
*   Removed the redundant call to `PasswordCipher.encrypt()` and the explicit memory scrub within the generator, as the caller (the UI and eventually `IdentityStorageManager`) is responsible for managing that memory lifecycle. 
*   `IdentityStorageManager.saveIdentity` now correctly receives raw bytes and performs the single, definitive encryption pass before writing to `EncryptedSharedPreferences`.

The `./gradlew test lint` executed successfully, passing all 72 tests.
